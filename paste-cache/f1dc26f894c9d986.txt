#1 - Dynamic Stripe Checkout for Dev E2E Testing
What & Why
The current checkout flow uses a static Stripe Payment Link (VITE_STRIPE_PAYMENT_URL) whose success/cancel redirect URLs are hardcoded in Stripe's dashboard. In dev, after completing payment, Stripe redirects back to the production app — breaking the full signup → payment → return loop in the dev environment.

The fix is to replace the static link with a server-side dynamic Stripe Checkout Session. The backend creates the session at request time, setting the success_url and cancel_url from the current request's origin. This means the redirect always goes back to the right environment — dev or production — with no Stripe dashboard changes needed.

The welcome email's "Sign-In Now" link (pointing to /login) does not need to change. It is already environment-aware (derived from the request host or APP_BASE_URL), and users who click it before completing payment will be correctly routed through the SubscriptionGuard to checkout.

Done looks like
After signup, a dev user clicks "Continue to secure checkout" on the /start page and is redirected to a real Stripe Checkout page (using test-mode Stripe credentials in dev).
After completing payment using a Stripe test card (e.g. 4242 4242 4242 4242), they are redirected back to the dev app's /stripe-success page with a valid session_id.
The /stripe-success page successfully attaches the subscription to their account.
The user can then log in and access the app with an active subscription.
If the user cancels on Stripe, they are returned to the /start page.
The static VITE_STRIPE_PAYMENT_URL env var is no longer required for the checkout flow to work.
All tests described below pass.
Out of scope
Changing the welcome email "Sign-In Now" link (already correct).
Changing the webhook handler or subscription management logic.
Changing the attach-session endpoint validation logic.
Stripe webhook forwarding setup for local dev (the attach-session flow handles subscription activation without webhooks).
Any changes to the production Stripe account or products.
Tasks
Implementation
Add POST /api/stripe/create-checkout-session backend route — Requires auth. Creates a Stripe Checkout Session in subscription mode using STRIPE_PRICE_ID, sets success_url to {origin}/stripe-success?session_id={CHECKOUT_SESSION_ID} and cancel_url to {origin}/start, pre-fills customer_email from the logged-in user's email, and sets client_reference_id to the user ID. Returns the session URL. Derive origin from APP_BASE_URL env var first, then fall back to the request's x-forwarded-proto/x-forwarded-host headers (matching the existing getBaseUrl pattern in authRoutes.ts). Return 503 if Stripe is not configured, 400 if STRIPE_PRICE_ID is not set.

Update start.tsx checkout button — Replace the static VITE_STRIPE_PAYMENT_URL redirect with a call to POST /api/stripe/create-checkout-session. On success, redirect to the returned session URL. Show a loading state while the session is being created. Display a clear error message if the request fails. Do not remove the Stripe redirect from the SubscriptionGuard component — that component uses VITE_STRIPE_PAYMENT_URL separately and is out of scope.

Add a dev setup guide section to CLAUDE.md — Document the env vars needed for local Stripe E2E testing (STRIPE_SECRET_KEY with a sk_test_... key, STRIPE_PRICE_ID with the test-mode price ID, STRIPE_WEBHOOK_SECRET optional for webhook events) and the test card to use (4242 4242 4242 4242, any future expiry, any CVC).

Unit Tests (server)
Unit test POST /api/stripe/create-checkout-session — Cover all of the following cases by mocking the Stripe SDK:
Returns 401 when the user is not authenticated.
Returns 503 when Stripe is not configured (no STRIPE_SECRET_KEY).
Returns 400 when STRIPE_PRICE_ID env var is not set.
Creates a Stripe Checkout Session with mode: 'subscription'.
Checkout session uses the configured STRIPE_PRICE_ID.
success_url contains /stripe-success?session_id={CHECKOUT_SESSION_ID}.
cancel_url contains /start.
customer_email is set to the logged-in user's account email.
client_reference_id is set to the logged-in user's ID.
origin is taken from APP_BASE_URL env var when set.
origin is derived from x-forwarded-proto + x-forwarded-host request headers when APP_BASE_URL is not set.
Returns the Stripe session URL on success.
Integration Tests (server)
Integration test the full session creation → attach flow — Using Stripe test-mode credentials (skip if STRIPE_SECRET_KEY not set in the test environment). Verify that a checkout session created by create-checkout-session can be attached via attach-session once the session is marked complete in Stripe's test API. Verify that existing attach-session behavior is unchanged.
Regression Tests
Regression tests for existing Stripe and subscription behavior — Verify none of the following are broken by the change:
POST /api/stripe/attach-session still validates session completeness, mode, and price ID correctly.
Admin role bypass (status: 'admin_bypass') still returns active: true from /api/me/subscription.
Dev subscription bypass (DEV_SUBSCRIPTION_BYPASS=true + ALLOW_UNSAFE_DEV_FEATURES=true) still returns active: true for non-admin users.
The welcome email signInUrl still points to {baseUrl}/login (no change to authRoutes.ts).
E2E Tests (browser)
E2E test the full signup → payment → access flow — Using Playwright against the running dev app with Stripe test credentials:
User signs up → is redirected to /start.
Clicking "Continue to secure checkout" calls the backend and redirects to a Stripe-hosted checkout page (verify URL is checkout.stripe.com).
Completing payment with test card 4242 4242 4242 4242 redirects back to /stripe-success?session_id=... on the dev app.
The /stripe-success page shows the success message and a "Get Started" button.
Clicking "Get Started" navigates to /welcome.
Verify the subscription is now active by calling /api/me/subscription and confirming active: true.
Cancel flow: clicking cancel/back on Stripe redirects to /start.
