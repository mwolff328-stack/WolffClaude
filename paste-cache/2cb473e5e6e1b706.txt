The fix is confirmed working. Here's a summary of what I found and what I changed:

Root cause: The portfolio/week endpoint returns entries: [] because the CMEA algorithm hasn't processed this pool's picks — but the actual picks do exist at a different endpoint (/entries/{id}/picks). The code was throwing a "no pick data" error and silently falling back to static mock data.

There was also a secondary bug: React's strict mode in development runs effects twice, and a hasFetched ref was preventing the second run from ever completing (the first run always got cancelled).

What I changed:

New nflTeams.ts — static map of NFL team abbreviations to full names, used to resolve team names from picks data

New types (EntryRecord, EntryPick) and two new API functions (fetchPoolEntries, fetchEntryPicks) to fetch real pick data from the entries endpoints

useSeasonData.ts — added a second-tier data path:

If portfolio/week returns empty entries → fetch entries list, then fetch each entry's picks in parallel
Build the 18-week view from those real picks (team, win/loss, alive entry count per week)
Only keeps static mock data if both paths completely fail
Fixed the StrictMode cancellation bug by resetting the hasFetched ref in the cleanup
The real data now loads in about 1 second. Weeks 1–4 show 2–3 alive entries with their actual picks (PHI, JAX, DEN in week 1, etc.), and week 5+ shows a single surviving entry — matching what actually happened in this pool.
